# 제2절 보안기능 — 2026 재구성

이 파일은 KISA 「Python 시큐어코딩 가이드」(2023)의 제2절을 2026년 7월 기준으로 교체한 정본이다. KISA 공식 2026 개정본이 아니며, 현재 Python·OWASP·NIST 자료를 결합한 재구성 기준이다.

## 목차

1. 적절한 인증 없는 중요 기능 허용
2. 부적절한 인가
3. 중요한 자원에 대한 잘못된 권한 설정
4. 취약한 암호화 알고리즘 사용
5. 암호화되지 않은 중요정보
6. 하드코드된 중요정보와 비밀 관리
7. 충분하지 않은 키 길이 사용
8. 적절하지 않은 난수 값 사용
9. 취약한 패스워드 허용
10. 부적절한 전자서명 확인
11. 부적절한 인증서 유효성 검증
12. 쿠키와 세션을 통한 정보 노출
13. 주석문·로그·진단자료의 주요정보
14. 부적절한 비밀번호 해시
15. 무결성 검사 없는 코드 다운로드
16. 반복된 인증시도 제한 기능 부재

## 공통 판정 원칙

- 인증, 인가, 소유권, tenant 격리와 민감 작업 재인증을 별도 통제로 확인한다.
- 보안 결정 실패는 기본 거부한다. 예외·timeout·키 조회 실패 때 보호 기능을 우회하지 않는다.
- 암호 알고리즘만 보지 말고 키 생성·저장·접근·회전·폐기와 nonce 수명주기를 함께 확인한다.
- 프레임워크 기본 보호를 유지하고 비활성화 옵션과 예외 경로를 우선 추적한다.
- 테스트 환경의 완화 설정이 운영으로 전파되지 않는지 확인한다.

## 1. 적절한 인증 없는 중요 기능 허용 (CWE-306)

계정 변경, 결제, 비밀 조회, 관리 기능, 데이터 내보내기 같은 중요 기능에 인증이 없거나 일부 경로에서 빠지면 공격자가 직접 호출할 수 있다.

### 안전한 코딩기법

- 라우터·미들웨어·dependency 등 일관된 경계에서 인증을 강제한다.
- background job, WebSocket, GraphQL resolver, 내부 API와 alternate HTTP method도 동일하게 보호한다.
- 비밀번호·MFA·복구수단 변경과 고위험 작업에는 최근 인증 또는 단계적 인증을 요구한다.
- 계정 잠금·탈퇴·권한 변경 시 기존 세션과 토큰을 적절히 무효화한다.
- 인증 실패를 캐시·예외 처리 때문에 성공으로 바꾸지 않는다.

익명 사용자, 만료·철회 토큰, 다른 인증 scheme과 직접 URL 호출을 테스트한다.

## 2. 부적절한 인가 (CWE-285, CWE-639)

인증된 사용자가 자신의 권한을 넘어 다른 사용자·tenant의 객체나 관리 기능에 접근하면 수평·수직 권한 상승이 발생한다.

### 안전한 코딩기법

- 모든 조회·수정·삭제에서 현재 주체의 owner·tenant·role 범위를 서버 측 쿼리에 포함한다.
- UI에서 버튼을 숨기는 것을 인가로 간주하지 않는다.
- object ID, field-level update, bulk operation, export와 간접 참조까지 확인한다.
- 정책은 deny-by-default로 정의하고 역할명보다 구체적 행위와 자원 관계로 판정한다.
- 모델·에이전트가 승인한 결과를 서버 측 인가 대신 사용하지 않는다.

```python
document = get_object_or_404(
    Document.objects.filter(owner=request.user),
    pk=document_id,
)
```

다른 owner·tenant의 유효한 ID와 낮은 권한 사용자를 이용한 회귀 테스트를 작성한다.

## 3. 중요한 자원에 대한 잘못된 권한 설정 (CWE-732)

파일, 소켓, DB, 클라우드 역할, 컨테이너 또는 CI 토큰에 과도한 권한을 부여하면 애플리케이션 침해의 영향이 확대된다.

### 안전한 코딩기법

- 프로세스·DB 계정·서비스 계정·배포 토큰에 최소 권한을 적용한다.
- 비밀·키·설정 파일의 소유자와 모드를 명시하고 `0o777` 같은 광범위 권한을 피한다.
- 업로드·임시·캐시 디렉터리에 실행 권한과 불필요한 공유 쓰기 권한을 제거한다.
- 컨테이너를 비루트로 실행하고 불필요한 capability·host mount·privileged 모드를 제거한다.
- CI job과 외부 PR이 production secret·publish 권한을 얻지 못하게 분리한다.

운영 배포 후의 실제 UID, umask, volume, IAM과 네트워크 정책을 확인한다.

## 4. 취약한 암호화 알고리즘 사용 (CWE-327, CWE-329)

DES·RC4·ECB, 고정 IV, 인증 없는 CBC 또는 자체 암호 설계는 기밀성·무결성을 보장하지 못한다.

### 안전한 코딩기법

- 검증된 라이브러리의 AEAD(AES-GCM, ChaCha20-Poly1305 등)를 우선한다.
- nonce를 키마다 재사용하지 않고 라이브러리 권고 길이와 생성 방식을 따른다.
- 암호화와 MAC 조합, padding, 키 파생을 직접 설계하지 않는다.
- 알고리즘 agility가 필요한 경우 서버 정책으로 허용 집합과 폐기 일정을 관리한다.
- 규제·상호운용 요구가 있으면 해당 프로파일과 운영 모드를 확인한다.

AES-CBC에 임의 IV만 추가한 구현을 일반적인 안전 예제로 인정하지 않는다.

## 5. 암호화되지 않은 중요정보 (CWE-312, CWE-319)

비밀번호, 토큰, 개인정보, 키와 업무 민감정보가 평문 저장·전송되면 파일·백업·네트워크·진단 경로에서 노출될 수 있다.

### 안전한 코딩기법

- 외부 및 내부 서비스 통신에 검증된 TLS를 사용한다.
- 저장 데이터는 위협 모델과 키 분리 요구에 따라 애플리케이션·DB·스토리지 암호화를 선택한다.
- 비밀번호는 복호화 가능한 암호화가 아니라 전용 password KDF로 저장한다.
- 백업, 메시지 큐, 캐시, 검색 인덱스, trace와 crash dump까지 데이터 흐름을 확인한다.
- 불필요한 민감정보를 수집·복제·보존하지 않는다.

## 6. 하드코드된 중요정보와 비밀 관리 (CWE-259, CWE-321, CWE-798)

코드·이미지·설정·테스트 fixture에 비밀번호, API 키, 토큰 또는 암호키를 넣으면 저장소와 빌드 아티팩트를 통해 노출된다.

### 안전한 코딩기법

- 관리형 secret store 또는 OS·오케스트레이터의 비밀 주입 기능을 사용한다.
- 접근 주체·환경·용도별로 비밀을 분리하고 최소 범위, 만료, 회전과 감사 기록을 관리한다.
- 환경변수도 프로세스 목록, 진단 덤프와 CI 로그에 노출될 수 있음을 고려한다.
- 비밀 값을 로그·오류·trace·메트릭 label에 기록하지 않는다.
- 유출이 의심되면 Git에서 삭제하는 데 그치지 말고 즉시 폐기·회전한다.

설정 파일과 같은 위치의 키로 해당 설정을 복호화하는 구조는 신뢰 경계를 분리하지 못한다.

## 7. 충분하지 않은 키 길이 사용 (CWE-326)

짧거나 폐기 예정인 키·파라미터는 계산 능력 증가와 암호 분석에 의해 보호 수준이 낮아진다.

### 안전한 코딩기법

- 현재 표준·라이브러리·조직 정책이 권고하는 알고리즘과 키 크기를 따른다.
- RSA는 신규 용도에서 최소 2048비트 이상을 기준으로 요구 수명과 규제 기준을 확인한다.
- 검증된 곡선과 파라미터를 사용하고 임의 곡선·DH group을 만들지 않는다.
- 키 수명, 데이터 보호 기간과 양자내성 전환 요구를 별도로 평가한다.
- 짧은 기존 키의 발견, 교체와 재암호화 계획을 마련한다.

## 8. 적절하지 않은 난수 값 사용 (CWE-330)

`random`과 예측 가능한 seed로 세션 ID, 토큰, OTP, nonce 또는 키를 만들면 공격자가 값을 추측할 수 있다.

### 안전한 코딩기법

- 보안 토큰은 `secrets` 또는 암호 라이브러리의 CSPRNG로 생성한다.
- 충분한 entropy와 인코딩 후 길이를 확보한다.
- 토큰은 목적·사용자·수명에 바인딩하고 일회용 또는 철회 가능하게 설계한다.
- 비교에는 필요할 때 constant-time API를 사용한다.
- 고유성과 예측 불가능성을 혼동하지 않는다. UUID 종류와 사용 목적을 확인한다.

```python
import secrets

reset_token = secrets.token_urlsafe(32)
```

## 9. 취약한 패스워드 허용 (CWE-521)

짧거나 유출된 비밀번호, 과도하게 예측 가능한 복구 절차는 계정 탈취 가능성을 높인다.

### 안전한 코딩기법

- 충분한 최소 길이와 긴 passphrase를 허용하고 불필요한 최대 길이 축소를 피한다.
- 유출·일반 비밀번호 목록을 차단한다.
- 붙여넣기와 password manager 사용을 방해하지 않는다.
- 정기적 강제 변경보다 침해·의심 이벤트 때 변경하고 재사용을 줄인다.
- MFA, 안전한 계정 복구, 재인증과 세션 무효화를 함께 설계한다.

클라이언트 검사와 동일한 정책을 서버에서 강제하고 Unicode 정규화 정책을 일관되게 적용한다.

## 10. 부적절한 전자서명 확인 (CWE-347)

서명 검증 생략, 공격자 선택 알고리즘, 잘못된 키 또는 검증 결과 무시는 데이터·토큰·업데이트 위조를 허용한다.

### 안전한 코딩기법

- 허용 알고리즘과 키 종류를 서버 설정에서 고정한다.
- 서명뿐 아니라 signer identity, 목적, issuer, audience, 만료, key ID와 revocation을 확인한다.
- JWT는 `iss`, `aud`, `exp`, 필요 시 `nbf`, `jti`와 키·알고리즘 혼동을 검증한다.
- 검증 실패·키 조회 실패·알 수 없는 알고리즘은 거부한다.
- 원본과 서명, 기대 digest가 같은 비신뢰 위치에서 함께 오지 않게 한다.

서명은 지정된 신원이 바이트를 승인했다는 근거이지 콘텐츠가 안전하다는 보증이 아니다. 서명 확인 후 동적 코드를 `eval`하는 구조를 안전하다고 보지 않는다.

## 11. 부적절한 인증서 유효성 검증 (CWE-295)

`verify=False`, `CERT_NONE`, 호스트명 검사 해제 또는 잘못된 신뢰 저장소는 중간자 공격을 허용한다.

### 안전한 코딩기법

- `ssl.create_default_context()`와 HTTP 클라이언트의 기본 인증서 검증을 유지한다.
- CA 체인과 호스트명을 모두 검증한다.
- 사설 CA가 필요하면 해당 CA만 제한적으로 추가하고 운영 배포를 관리한다.
- IP 접속 시 인증서 식별자와 SNI 정책을 확인한다.
- connect/read/write timeout과 최대 응답 크기를 설정한다.
- pinning은 회전·장애 대응 요구를 갖춘 경우에만 사용한다.

## 12. 쿠키와 세션을 통한 정보 노출 (CWE-539, CWE-614, CWE-1004, CWE-1275)

민감정보를 쿠키 payload에 저장하거나 보안 속성·세션 회전·무효화가 없으면 탈취와 세션 고정 위험이 커진다.

### 안전한 코딩기법

- 쿠키에는 opaque session ID만 저장하고 민감정보를 직접 넣지 않는다.
- HTTPS에서 `Secure`, JavaScript 접근이 불필요하면 `HttpOnly`, 요청 모델에 맞는 `SameSite`를 설정한다.
- Domain·Path와 수명을 최소화한다.
- 로그인·권한 변경 시 session ID를 회전하고 로그아웃·비밀번호 변경·계정 잠금 때 무효화한다.
- 서버 측 세션에 절대·유휴 만료, 동시 세션 정책과 철회 기능을 둔다.

서명된 쿠키는 기밀성을 제공하지 않으며 장기 토큰에는 회전·철회 전략이 필요하다.

## 13. 주석문·로그·진단자료의 주요정보 (CWE-532, CWE-615)

주석, 샘플, 로그, exception, trace와 디버그 화면에 자격증명·내부 경로·개인정보가 남으면 저장소와 관측 시스템을 통해 노출된다.

### 안전한 코딩기법

- 비밀, 실제 계정, 내부 주소와 운영 데이터를 주석·예제에 넣지 않는다.
- 로그 필드 허용 목록과 마스킹 정책을 사용한다.
- 요청·응답 전체, Authorization, Cookie, reset token을 기본 기록하지 않는다.
- trace·APM·crash dump의 접근·보존·삭제 정책을 설정한다.
- secret scanning을 pre-commit·CI와 저장소 이력에 적용하되 탐지 결과를 즉시 회전 절차와 연결한다.

## 14. 부적절한 비밀번호 해시 (CWE-759, CWE-916)

MD5·SHA 계열 같은 빠른 일반 해시 또는 고정 salt는 오프라인 추측 공격에 취약하다.

### 안전한 코딩기법

- 프레임워크의 password hasher 또는 Argon2id·scrypt·bcrypt·PBKDF2 같은 적응형 password KDF를 사용한다.
- 라이브러리가 salt를 생성하게 하고 비용 값은 운영 환경에서 측정·조정한다.
- pepper가 필요하면 비밀번호 DB와 분리된 비밀 저장소에 둔다.
- 알고리즘과 비용을 저장해 로그인 성공 시 점진적으로 재해시한다.
- 비교·오류 응답으로 사용자 존재나 해시 상태를 노출하지 않는다.

SHA-256에 salt를 직접 붙이는 예제를 안전한 비밀번호 저장으로 인정하지 않는다.

## 15. 무결성 검사 없는 코드 다운로드 (CWE-494)

네트워크에서 받은 코드·패키지·업데이트를 출처와 무결성 확인 없이 실행하면 공급망 공격으로 이어질 수 있다.

### 안전한 코딩기법

- 실행 코드는 검토·빌드된 정적 아티팩트로 배포한다.
- 허용 저장소, signer identity, 예상 digest, 인증서·키 유효기간과 폐기를 정책으로 검증한다.
- 검증 실패 시 실행하지 않는다.
- 최소 권한, 격리, provenance, 단계적 배포와 롤백을 함께 사용한다.
- 다운로드한 문자열을 서명 확인 후 `eval`하지 않는다.

해시 파일을 아티팩트와 동일한 비신뢰 채널에서 함께 받는 것만으로 출처가 확인되지 않는다.

## 16. 반복된 인증시도 제한 기능 부재 (CWE-307)

로그인·MFA·복구코드·비밀번호 재설정에 시도 제한이 없으면 brute force와 credential stuffing이 가능하다.

### 안전한 코딩기법

- 계정, IP·네트워크, device와 위험 신호를 조합한 rate limit을 적용한다.
- 점진적 지연과 임시 잠금을 사용하되 공격자가 특정 계정을 영구 차단하지 못하게 설계한다.
- 성공·실패 응답과 처리 시간을 유사하게 유지해 계정 열거를 줄인다.
- MFA·CAPTCHA를 위험 기반 보조 통제로 사용한다.
- 실패율·분산 시도·새 위치·토큰 재사용을 탐지하고 경보·대응 절차에 연결한다.

분산 환경에서 limiter 저장소의 원자성, 우회 경로와 장애 시 동작을 테스트한다.

## 검증 체크리스트

- [ ] 익명, 낮은 권한, 다른 owner·tenant, 만료·철회 토큰을 테스트한다.
- [ ] 인증·인가·서명·인증서 실패가 fail closed인지 확인한다.
- [ ] 키·비밀의 생성, 저장, 주입, 접근, 회전과 폐기를 확인한다.
- [ ] 쿠키·세션·JWT의 수명, 회전, 철회와 검증 claim을 확인한다.
- [ ] 비밀번호 정책·해시 비용과 인증시도 제한을 운영 환경에서 검증한다.
- [ ] 로그·trace·백업·빌드 아티팩트에 민감정보가 없는지 확인한다.
